Section 1

Governance & Hybrid SDLC Overview

Water-Scrum-Fall model · SBV Compliance Pillars

This bank operates a Hybrid SDLC model — colloquially referred to as Water-Scrum-Fall — which strategically combines the iterative velocity of Agile/Scrum for software execution with the structured oversight of Waterfall for project governance and production releases. This approach satisfies both business agility requirements and the non-negotiable compliance mandates of the State Bank of Vietnam (SBV).

🔭 Waterfall — Initiation
Formal business case, governance approval gates, budget allocation (Tờ trình), and regulatory sign-off before any development begins.
⚡ Agile — Execution
Iterative Sprints, continuous integration, Feature team backlogs, and Scrum ceremonies to deliver working software incrementally.
🔐 Waterfall — Release
Sequential Security & Performance testing gates, CAB review, mandatory SBV audit checkpoints, and controlled production deployment.

🛡️ Vietnamese Banking Compliance Pillars

As mandated by SBV Circular 09/2020/TT-NHNN & internal policy

No single individual may hold conflicting roles across development, testing, and production deployment. Enforced by role-based access controls (RBAC) in GitLab, JIRA, and the production deployment pipeline.

ℹ️
Developers cannot merge their own code to the main branch. Testers must be independent of the development team. Release Managers hold the sole authority to initiate production deployments.

All systems handling financial transactions or customer PII must pass a full security assessment cycle before every production release, including:

  • 1 Static Application Security Testing (SAST) — automated scan on every commit.
  • 2 Dynamic Application Security Testing (DAST) — executed in the SIT/UAT environment.
  • 3 Penetration Testing — performed by the independent Internal Security Tester team.
  • 4 Vulnerability Assessment Report — must be signed off before CAB submission.
⚠️
Any CVSS score ≥ 7.0 (High) identified during security testing constitutes a hard blocker. The release cannot proceed until remediated and re-tested.

The CAB is the final governance authority before any change reaches production. Membership includes Technology Leadership, Risk Management, Compliance, and Business Owners.

📅
CAB Meeting Frequency
Weekly, every Tuesday at 14:00 ICT. Emergency CABs convened within 4 hours for P1 issues.
📄
Change Request (CR)
Must include PT Report, Security Report, UAT sign-off, rollback plan, and deployment timeline.
🗳️
Voting Quorum
Minimum 5 of 7 CAB members must be present. Unanimous approval required for Tier 1 systems.
⏳
Freeze Windows
No releases during year-end banking freeze (Dec 20 – Jan 5) or SBV reporting periods.

Waterfall governance · Planning & approval gates

🔄 Initiation Phase Gate Sequence

  • Gate 0 · Concept & Ideation
    Business pain point identified; initial scope scoped; Executive Sponsor nominates Initiative Director.
  • Gate 1 · Business Case Approved
    BOD or Delegated Authority reviews and approves the Business Case and Tờ trình. Budget is formally allocated.
  • Gate 2 · Project Charter Signed
    Project Manager appointed; Team assembled; Project Charter countersigned by Sponsor, Initiative Director, and Business Owner.
  • Gate 3 · Architecture Review Board (ARB)
    Enterprise Architect presents solution design. ARB validates compliance with bank technology standards and SBV IT regulations.
  • Gate 4 · Development Kick-off
    Project Plan approved. Scrum teams onboarded. JIRA Epic structure created. Sprint 0 begins.

✅ Definition of Done (DoD) — Sprint Level

All criteria must be satisfied before a Story or Epic can be considered complete.

  • 1Code peer-reviewed and approved by minimum 2 reviewers via GitLab Merge Request (no self-merge).
  • 2Unit test coverage ≥ 85% as validated by SonarQube gate. No critical or blocking code smells.
  • 3SAST scan passed — zero High / Critical severity findings in the feature branch.
  • 4Functional tests passing in SIT environment; smoke tests passing in UAT environment.
  • 5API contract tests passing (if applicable). No regression in existing test suites.
  • 6Acceptance criteria validated and sign-off obtained from Product Owner.
  • 7JIRA story updated to "Done", linked to the relevant commit SHA and test results artifact.

👤 View by Role — Duties & Responsibilities

📋
Release Manager
Owns the end-to-end release lifecycle. Single point of accountability for production deployment.

Pre-Deployment Duties:

  • Create and manage the Release Request in JIRA with full scope definition and changelog.
  • Coordinate environment provisioning with DevOps for PERF and SEC testing environments.
  • Confirm artifact build integrity (SHA checksum) and deployment package completeness.
  • Track gate-passing status across all 12 pipeline steps; escalate blockers immediately.
  • Compile the Change Request (CR) package: PT Report, Security Report, UAT sign-off, Impact Analysis, Rollback Plan.
  • Present the Change Request at the CAB meeting; answer all board queries.
  • Conduct the Go/No-Go call with DevOps, Business Owner, and on-call support before PROD deployment.
  • Publish release notes and notify all stakeholders upon successful deployment.
  • Manage the 72-hour post-release watch period and coordinate the Lessons Learned session.
📌
The Release Manager is the only role authorized to transition the pipeline between Steps 1→2, 5→6, and 10→11. These transitions require explicit sign-off entries in the GRC system.
📊
Performance Tester
Responsible for validating system throughput, latency, and stability under peak banking load conditions.

Performance Testing Duties:

  • Confirm PERF environment topology matches PROD specifications (CPU, RAM, DB replication config).
  • Design and execute Load Test: simulate peak concurrent users (baseline from production metrics).
  • Execute Stress Test: push system to 150% of expected peak load to identify breaking point.
  • Execute Soak Test: sustain load for minimum 4 hours to detect memory leaks or connection pool exhaustion.
  • Capture KPIs: Transactions Per Second (TPS), P95/P99 latency, error rate, CPU/memory utilization.
  • Validate all KPIs against SLA thresholds defined in the Performance Test Plan.
  • Generate the Performance Test Report using the approved bank template, including Grafana/JMeter screenshots.
  • Obtain sign-off from the Performance Test Lead and submit report to Release Manager.
  • Transition pipeline gate from "In Performance Testing" → "PT Done" in JIRA.
⚠️
If any KPI fails SLA thresholds (e.g., P95 latency > 2000ms for core banking APIs), the release is immediately blocked. Development team is notified for performance remediation before re-testing.
🔐
Security Tester
Independent security team responsible for identifying vulnerabilities before production exposure.

Security Testing Duties:

  • Review SAST findings from CI pipeline; confirm all critical/high findings are remediated by Dev team.
  • Execute DAST scan (Dynamic Application Security Testing) using OWASP ZAP against the SEC environment.
  • Conduct manual penetration testing covering OWASP Top 10 and banking-specific threat vectors (e.g., parameter manipulation, session hijacking, privilege escalation).
  • Assess authentication & authorization controls: MFA, RBAC enforcement, token expiry.
  • Review data encryption at rest and in transit (TLS 1.2+ enforced, AES-256 for PII fields).
  • Validate API security: rate limiting, input validation, SQL injection, XSS prevention.
  • Score all findings using the CVSS v3.1 framework; classify by Severity and Business Impact.
  • Generate the Security Assessment Report; obtain CISO sign-off before releasing to Release Manager.
  • Actively monitor the 72-hour post-release period for anomalous security events in SIEM.
🚨
Hard Blocker Rule: Any finding with CVSS ≥ 7.0 (High) or any finding in OWASP Top 10 Category A01-A05 is an automatic deploy blocker. The CISO has veto power over the CAB regarding security findings.

Phase 4 — Deployment & Commercialization

Production handover · Ops transition · Monitoring & support

🚀
Milestone: CAB has approved the Change Request. The deployment window is confirmed. This phase covers the controlled production release and the formal handover from the Project Team to the permanent Operations & Customer Support team.

📦 Key Deployment Outputs

  • 📘
    Operations Handover Document
    System architecture, runbook, escalation matrix, and SLA definitions
  • 📗
    User & Admin Guides
    End-user training materials, admin console guide, and onboarding flow
  • 📊
    Monitoring Dashboard
    Grafana/Datadog real-time metrics, alerting rules, and SLO tracking
  • 🚨
    Incident Management Process
    Severity classification, on-call rotation, escalation path, and war-room protocol
  • 📋
    Technical & End-User FAQs
    Known issues, workarounds, and L1/L2/L3 support triage guide
  • 📈
    Deployment Execution Tracker
    Step-by-step deployment log, timestamps, and post-release verification results

🤝 Project → Operations Handover

  • Go/No-Go call (T-2h)
    All parties (RM, DevOps, Business Owner, L1 Support Lead) confirm readiness on a bridge call.
  • PROD Deployment Executed
    DevOps runs deployment pipeline in the approved maintenance window (typically 22:00-02:00 ICT).
  • Smoke Test & Verification
    Release Manager executes smoke test checklist. Business Owner validates core transaction flows in PROD.
  • Handover Meeting
    Formal handover meeting with Ops Lead. All artifacts transferred. Access credentials handed off. On-call duties activated.
  • 72-Hour Watch Period
    Dev team on standby. Monitoring dashboards reviewed every 4 hours. Any P1/P2 incident triggers immediate war-room.
  • Project Closure
    Lessons Learned session. PIR (Post-Implementation Review) report published. Project formally closed in PPM tool.

📊 Monitoring & Incident Response Framework

🟢
P4 — Low
Cosmetic issue. No business impact. Response: Next business day. Resolved in weekly sprint.
🔵
P3 — Medium
Minor disruption. Workaround available. Response: 4 hours. L2 Support assigned.
🟡
P2 — High
Significant impact. No workaround. Response: 1 hour. L3 + Dev team engaged.
🔴
P1 — Critical
Core banking down. Response: 15 min. War-room activated. CPTO & CRO notified. Rollback considered.

📁 Commercialization Artifacts Reference

The Operations Handover Document (OHD) is the primary artifact that enables the Operations team to independently manage the system post-deployment.

  • ASystem architecture diagram and component inventory.
  • BInfrastructure and deployment configuration (Kubernetes, Terraform manifests).
  • CDatabase connection strings, secrets management (Vault paths — no plaintext credentials).
  • DMonitoring thresholds and alert runbook.
  • EEscalation matrix: L1 → L2 → L3 → Vendor support contacts.
  • FBackup and recovery procedures with tested RTO/RPO targets.

Real-time observability stack: Grafana for metrics visualization, Loki for log aggregation, Jaeger for distributed tracing, and PagerDuty for on-call alerting.
