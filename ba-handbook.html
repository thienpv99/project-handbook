Introduction

Purpose & Scope

Who should read this · What it covers · How to use it
What This Handbook Is

This handbook defines the Business Analyst (BA) workflow for all software implementation projects at the bank. It documents BA responsibilities, deliverables, interaction points, and quality gates across every phase of project execution — from initial kickoff through post-implementation review.

It is designed to be read by the entire project team — not just BAs — so that every stakeholder understands what to expect from the BA function, when to involve them, and how to collaborate effectively.

👩‍💼
Primary Audience
Business Analysts · Project Managers · Solution Architects · QA Leads · Product Owners
📖
Secondary Audience
Development Teams · Business Stakeholders · Compliance Officers · UX/UI Designers

Governing Framework
BABOK® Guide v3 · IEEE 830 · ISO/IEC 25010 · SBV Circular 09/2020/TT-NHNN
ℹ️
How to use this handbook: Navigate by phase using the sidebar. Each phase states its trigger, activities, primary outputs, and how it contributes to adjacent team workflows (PM, SA, QA, DEV). Cross-reference with the Deployment Runbook for pre-production and go-live procedures.
⚠️
SDLC Context: The bank operates a Hybrid Water-Scrum-Fall model. Core/stable requirements follow Waterfall governance (formal sign-off, RTM, CCB). Evolving feature requirements are managed as an Agile backlog within sprints. This handbook covers both tracks — look for the [WATERFALL] [AGILE] [HYBRID] tags in each phase.

Phase 0 · Waterfall Gate · Trigger: Project Kickoff

Project Initiation & Planning

Stakeholder Register · BA Plan · Business Need Statement · Compliance Scope
  1. 1
    Review Project Charter, SOW, or Business Case — understand scope boundaries, budget envelope, strategic alignment.
  2. 2
    Build Stakeholder Register — for each stakeholder: role, influence level, interest level, communication preference, RACI position.
  3. 3
    Conduct initial interviews — capture objectives, KPIs/OKRs, pain points. Do not propose solutions yet.
  4. 4
    Define Business Need Statement:
    "The [stakeholder] needs [capability] in order to achieve [outcome] because [rationale]."
  5. 5
    Establish BA Plan — elicitation methods, tools, review cadence, deliverables per phase, RACI for BA outputs.
  6. 6
    Identify compliance/regulatory scope early — SBV Circular 09/2020, Decree 13/2023, PCI-DSS if card data in scope.
  • 📋
    Stakeholder Register — role mapping with RACI and communication preferences
  • 📌
    Business Need Statement — one-sentence framing of the core problem/opportunity
  • 📅
    BA Plan — elicitation schedule, tool selection, deliverable owners, review cadence
  • ⚖️
    Compliance Scope Checklist — regulations in scope, regulatory contact, key obligations
📅
→ Project Manager
Stakeholder Register + Business Need Statement validate the Project Charter's objectives.
⚖️
→ Compliance Officer
Compliance Scope Checklist initiates early regulatory review, preventing late-stage rework.

Phase 1 · Hybrid · Trigger: BA Plan Approved

Elicitation & Collaboration

Interviews · Workshops · Observation · Assumptions Log
  1. 1
    Select elicitation technique based on stakeholder type, information complexity, and SDLC phase (see Technique Catalog).
  2. 2
    Capture raw notes without interpretation during sessions. Separate observation from inference.
  3. 3
    Classify inputs post-session into: Requirements / Constraints / Assumptions / Risks / Questions-to-follow-up.
  4. 4
    Confirm captured information with stakeholders — replay what you heard before moving to analysis.
  5. 5
    In Hybrid SDLC — Phase 1 is continuous during Agile sprint cycles for evolving requirements; for core Waterfall scope it is a discrete, time-boxed activity.
🚫
Anti-patterns to avoid in Phase 1:
❌ Leading questions ("You want X, right?")  ·  ❌ Solution-first framing (redirect to the problem)  ·  ❌ Relying on loudest stakeholder only  ·  ❌ Skipping confirmation of captured information
  • 📝
    Elicitation Notes — raw, uninterpreted session notes with date, attendees, method used
  • ❓
    Assumptions & Constraints Log — explicitly documented, owner-assigned, reviewed at Phase 3
  • 📊
    Interview Summaries — stakeholder-confirmed summaries of key insights per session
  • ⚠️
    Risk Register input — business risks surfaced during elicitation, fed to PM Risk Register
📐
Phase 2 · Hybrid · Trigger: Sufficient Elicitation Data

Requirements Analysis & Design Definition

BRD · SRS · User Stories · RTM · Business Rules · NFR Spec
  1. 1
    Classify by BABOK type — Business / Stakeholder / Solution (Functional + Non-Functional) / Transition.
  2. 2
    Model As-Is and To-Be using:
    • BPMN 2.0 — business processes
    • Use Case Diagrams / User Story Maps — scope
    • DFD / ERD — data flows and entities
    • State Transition Diagrams — workflow status
  3. 3
    Define Business Rules separately from functional requirements — avoid mixing rule logic with system behavior descriptions.
  4. 4
    Specify NFRs using ISO/IEC 25010 — every NFR must be measurable (see NFR Framework section).
  5. 5
    Write in SDLC-appropriate format:
    • [WATERFALL] — Shall-statements in BRD/SRS with unique IDs
    • [AGILE] — User Stories: "As a [role] I want [goal] so that [benefit]" with GWT acceptance criteria
  6. 6
    Prioritize using explicit framework — MoSCoW for Waterfall scope lock; WSJF for Agile backlog. Document the choice in BA Plan.
  7. 7
    Build and maintain RTM — links: Business Requirement → Solution Requirement → Test Case → Defect.
  8. 8
    Validate against SMART + INVEST criteria — see Quality Gates section.
  • 📘
    BRD / SRS — formal requirements document (Waterfall) or living specification
  • 📝
    User Stories with Acceptance Criteria — GWT format, backlog-ready
  • 🔀
    Process Flow Diagrams (As-Is / To-Be) — BPMN 2.0 notation
  • 📊
    Requirements Traceability Matrix (RTM) — BR → FR → TC linkage
  • 📖
    Data Dictionary & Business Rules Catalog — canonical definitions, rule owners
  • ⚡
    NFR Specification — measurable non-functional requirements per ISO 25010
  • 🎯
    Prioritized Backlog — ordered by chosen prioritization framework

Phase 3 · Waterfall Gate · Trigger: Draft-Complete Requirements

Requirements Validation & Approval

Formal sign-off · Baseline · Change Request process activation
  1. 1
    Conduct Requirements Review meeting — mandatory attendees: Business Stakeholders, PM, SA, QA Lead, Compliance Officer (if in scope).
  2. 2
    Log all feedback, update documents accordingly within agreed SLA.
  3. 3
    Obtain formal written sign-off — verbal approval is NOT acceptable for formal baseline.
  4. 4
    Baseline requirements — all subsequent changes route through the Change Request process (managed by BA, approved by CCB).
  5. 5
    Publish to shared repository — notify all consuming teams (DEV, QA, SA) of the baselined version.
🚫
Critical rule: Any scope change after baseline must go through the BA for Impact Assessment before PM updates the schedule or SA changes the design. Silent changes to requirements are a project risk.
🏗️
Phase 4 · Agile Sprint · Trigger: SA/DEV Begins Design

Solution Evaluation & Design Support

Design reviews · Wireframe validation · Interface specs · Data mapping
  1. 1
    Participate in Solution Design Reviews — verify design satisfies requirements, not just technical elegance.
  2. 2
    Review UI/UX wireframes and prototypes against stakeholder expectations. Provide structured feedback referencing specific AC.
  3. 3
    Validate Data Mapping for integration/migration features — ensure field mappings match Data Dictionary.
  4. 4
    Write Interface Specifications / API Requirements — input/output contracts, error handling, business rules for integrations.
  5. 5
    Support DEV estimation — provide business context. BA does not estimate effort but ensures stories are well-understood before Planning Poker.
UI Review Feedback Interface Specification Data Mapping Document Solution Trade-off Analysis
⚙️
Phase 5 · Agile Sprint · Trigger: Development Sprint Begins

Implementation Support

Clarifications · Change Requests · Defect triage · DoR/DoD gates
  1. 1
    Primary requirements clarifier for DEV — respond to clarification requests within agreed SLA (recommended: same business day).
  2. 2
    Manage Change Requests — conduct impact analysis (scope/timeline/cost), submit to PM + CCB for approval, update RTM post-approval.
  3. 3
    Review test cases against acceptance criteria — ensure QA is testing the right behaviors.
  4. 4
    Support defect triage — classify as: Requirement Defect (BA owns) vs. Implementation Defect (DEV owns).
  5. 5
    Maintain Definition of Ready gate — no story enters sprint without DoR checklist complete.
Change Request Log Impact Assessments
🧪
Phase 6 · Waterfall Gate · Trigger: DEV Declares UAT-Ready

User Acceptance Testing (UAT)

UAT Plan · Test Scripts · Defect Log · Written Sign-off
  1. 1
    Prepare UAT Plan — scope, objectives, entry/exit criteria, schedule, roles.
  2. 2
    Create UAT Test Scripts directly from acceptance criteria (GWT format maps 1:1 to test steps).
  3. 3
    Coordinate and facilitate UAT execution with business users — BA does not execute tests, business users do.
  4. 4
    Track and classify UAT defects — Critical (blocks go-live) / Major (workaround exists) / Minor (cosmetic).
  5. 5
    Obtain UAT Sign-off from authorized business stakeholders — must be written, not verbal.
UAT Plan UAT Scripts UAT Defect Log UAT Sign-off
🚢
Phase 7 · Waterfall · Trigger: UAT Passed, CAB Approved

Release & Transition

Release Notes · User Guide · Training · Rollback criteria
  1. 1
    Contribute to Release Notes — write the business impact section (what changed for users, not what changed in code).
  2. 2
    Validate Training Materials and User Guides — confirm accuracy against final requirements and UAT-confirmed behaviors.
  3. 3
    Define rollback/fallback criteria from a business perspective — what business conditions trigger a rollback decision.
  4. 4
    Support hypercare period — available for business user questions in the first 1–2 weeks post go-live.
Release Notes (business section) User Guide Training Deck Cutover Plan input
📊
Phase 8 · Review · Trigger: Stabilization Ends (2–4 weeks post go-live)

Post-Implementation Review

Benefits realization · Lessons learned · Next iteration seeding
  1. 1
    Benefits Realization Review — measure original KPIs/OKRs stated in Phase 0. Did the solution achieve what was promised?
  2. 2
    Document Lessons Learned — BA perspective: what elicitation techniques worked, what requirement issues emerged late, what to do differently.
  3. 3
    Update Business Process Documentation — the To-Be from Phase 2 becomes the new As-Is baseline for future projects.
  4. 4
    Identify backlog items for the next iteration — feed into Product Roadmap and seed Phase 0 of next cycle.
PIR Report Lessons Learned Log Updated Process Docs Product Roadmap input

Non-Functional Requirements Framework

Every NFR must be measurable — no ambiguous language
📐
Rule: Every NFR must be measurable. ❌ "The system shall be fast" is not an NFR. ✅ "P95 response time shall be < 500ms at 1,000 concurrent users" is an NFR.
⚡ Performance Efficiency
Response time · Throughput · Resource utilization
e.g., P95 API response < 500ms, 99.5% uptime SLA
🔒 Security
Authentication · Authorization · Encryption at rest/transit · Audit logging
Must comply with SBV 09/2020, Decree 13/2023
🛡️ Reliability
Availability (SLA%) · Fault tolerance · RTO / RPO
e.g., RTO < 4h, RPO < 1h for core banking systems
♿ Usability
Learnability · Accessibility (WCAG 2.1 AA minimum) · UX
e.g., new user completes core task within 3 minutes unassisted
🔧 Maintainability
Modularity · Testability · Modifiability
e.g., new feature deployable without system restart
🔄 Compatibility
Co-existence · Interoperability with other systems
e.g., compatible with CBS v14.x, ISO 20022 message format

Criteria that BA validates before requirements progress to next phase

Measurable

Achievable

Relevant

I
Independent
No hidden dependency on another story
N
Negotiable
Details can evolve through conversation
V
Valuable

E
Estimable

Testable

✅ Definition of Ready (DoR)
Gate before story enters sprint
  • User story follows "As a… I want… so that…" format
  • Acceptance criteria in Given-When-Then format
  • Dependencies identified and resolved or sequenced
  • UX mockups attached (if UI story)
  • NFRs specified or referenced
  • Business rules referenced
  • Estimated by DEV team
  • Small enough to complete within one sprint
🏁 Definition of Done (DoD)
Gate before story is accepted
  • All acceptance criteria met and verified
  • Code peer-reviewed and merged
  • Unit and integration tests pass
  • QA testing complete, no Critical/Major defects open
  • NFR validation passed (perf, security, accessibility)
  • Documentation updated (user guide, API docs)
  • Demonstrated and accepted by PO/Business
  • Deployed to staging environment

Anti-Patterns — What Not to Do

Common BA mistakes and the correct approach
❌
Ask leading questions — "You want the button on the top right, correct?" Instead: ask open-ended, neutral questions.
❌
Accept solutions as requirements — "I need a dropdown with 5 items." ✅ Redirect: "Before the solution, can you describe the business problem it solves?"
❌
Rely only on the loudest stakeholder — actively seek silent voices who may represent critical user groups.
❌
Skip confirmation of captured information — always replay what you heard before moving to analysis.
❌
Write requirements that dictate implementation — "using Kafka for event streaming." Requirements say WHAT, not HOW.
❌
Mix multiple requirements in one statement — use "and" as a smell indicator. Split compound requirements.
❌
Use ambiguous words: "user-friendly", "fast", "robust", "intuitive", "seamless." Every adjective must be measurable.
❌
Skip NFRs "because they'll figure it out in design" — undocumented NFRs become the most expensive defects to fix late.
❌
Accept verbal approval for formal baseline — written sign-off is mandatory for Waterfall and Hybrid core scope.
❌
Allow scope changes without Change Request process post-baseline — every change needs impact assessment before implementation.
❌
Update requirements silently without notifying QA and updating the RTM — test coverage gaps will follow.
❌
Act as a message relay between stakeholders and DEV — facilitate the conversation directly, don't play telephone.
